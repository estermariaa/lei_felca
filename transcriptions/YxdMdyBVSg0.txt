Na maior parte dos casos, os anti-sheets funcionam exatamente no chamado Ring Zero do Windows, uma camada de nível kernel do sistema. Assim, eles têm basicamente as permissões mais avançadas possíveis para manipular o sistema como eles quiserem. A alegação mais comum para se utilizar essa técnica, o rodar anti-sheet, é que se o software tiver um nível alto de permissões, eles têm maiores chances de parar quase qualquer tentativa de hack que tente alterar os arquivos do jogo ou se camuflam em arquivos do sistema. O Linux não costuma permitir que esse tipo de coisa aconteça, porque instalar um software dessa forma é visto como um problema de segurança. Então, talvez a resposta não seja fácil, mas pelo menos agora existe uma proposta recente chamada Multicaddle, ela apareceu literalmente mês passado. A ideia é utilizar essa tecnologia em Enterprise, como o grande parte das coisas desenvolvidas no entorno do Linux, mas isso pode ter uma aplicabilidade muito interessante para os jogos na plataforma. A estrutura de Multicaddle muda a arquitetura de um sistema Linux de uma forma bem interessante. Olha só, isso aqui é um sistema Linux de desktop tradicional com as suas camadas. No topo, nós temos os apps e os games, depois nós temos a interface, o shell e só então nós chegamos no kernel, que é o Linux de fato, na verdade. Geralmente o kernel é um componente único, mas a proposta de Multicaddle é justamente mudar isso e rodar mais de um kernel simultaneamente com um selector que modifique o kernel usado, dependendo da aplicação, sem precisar reiniciar o sistema. Isso pode ser usado para rodar aplicações específicas de forma otimizada, utilizando um kernel customizado. Encontro outras aplicações, rodam um outro kernel genérico, por exemplo. Realmente é muito útil em Enterprise, já que um único sistema poderia rodar aplicações normais e aplicações em real-time ao mesmo tempo. No caso dos desktops e dos jogos, nós poderíamos usar kernels para finalidades diferentes, como um kernel utilizado para jogos, ao lado de um kernel regular para as demais atividades, sem precisar compartilhar recursos de hardware, não vai gastar mais recursos por eles estarem rodando ao mesmo tempo. E aí, esticando um pouco a imaginação, poderia ver um kernel homologado pelos anti-sheets, que talvez pudesse ser o kernel do Steam OS até mesmo, e toda vez que a gente abrisse um game, esse kernel seria utilizado no lugar do kernel regular da disto. A parte mais interessante é que essa seria talvez a melhor solução contra hackers da indústria de jogos até então. Com um kernel homologado, cheio de hash comparativos para cada um dos arquivos que seja imodificável, qualquer software que tente alterar as características dele, como um cheat, automaticamente entraria em violação e poderia ser barrado. O jogo poderia até se negar a rodar se não estivesse rodando no kernel específico. Sem falar que essa abordagem reduziria consideravelmente os problemas de privacidade, já que essa estrutura só entra em ação enquanto você estiver jogando e não fica rodando o tempo inteiro mesmo com o jogo fechado. Sendo uma implementação open-source, mesmo que os módulos de anti-sheets sejam proprietários, ainda assim seria possível ser parcialmente auditado para que a segurança pudesse ser melhorada num esporço entre comunidade e empresas.
